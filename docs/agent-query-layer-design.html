gtmDB Query Agent

Complete Implementation Plan & Architecture

Overview & Vision

The Problem

External agents (Marketing, Sales) need to answer complex analytical questions like:

  • "What are my best-performing campaigns by LTV?"
  • "What's the conversion rate by channel?"
  • "Show me all touches on this deal, with their ROI impact"

Current approach: Rigid, pre-built endpoints for each metric. Doesn't scale. Can't handle new questions without API changes.

The Solution

An internal LangGraph-based AI agent that:

  • Receives intent-based natural language questions
  • Generates Cypher (Neo4j) and SQL (ClickHouse) queries on the fly
  • Combines graph traversal with OLAP analytics
  • Operates safely within permission boundaries (3-layer guard)
  • Self-heals on errors and retries with corrected queries

Architecture Layers

Layer 1: Soft Guidance

Agent's system prompt tells it what the caller can access. Agent self-limits.

Layer 2: Query Guard

Parses generated Cypher/SQL, validates against permission set, rejects violations before execution.

Layer 3: DB-Level

Read-only users, row policies. Physically impossible to bypass, even if Layers 1+2 fail.

Core Architectural Decisions

1. ClickHouse for Analytics (Not Neo4j Aggregation)

Decision

Use ClickHouse as a dedicated OLAP store for metrics, not Neo4j aggregations.

Why

  • Neo4j limitation: Large aggregations (>1M intermediate rows) hit the EagerAggregation bottleneck. Graph DBs are optimized for traversal, not aggregation.
  • ClickHouse advantage: Columnar architecture, naturally suited for GROUP BY/SUM/COUNT operations
  • Flexibility: Query any column independently without pre-planned indexes
  • Durability: Enterprise-grade, replicated, no data loss

Tradeoff

Need to maintain dual-write (Neo4j → ClickHouse sync) and migrate existing data. Initially direct inserts; later Kafka for async streaming.

2. Text2Cypher/Text2SQL Instead of Pre-Built Toolkit

Decision

Agent generates raw Cypher and SQL directly, not pre-built operation tools.

Before (Rigid)

  • ~10 pre-built tools: graph_explore, olap_aggregate, olap_funnel, etc.
  • Each tool = fixed parameters, limits flexibility
  • Adding new tool = new code
  • Defeats the purpose of having an agent (we wanted to escape rigidity!)

After (Flexible)

  • 3 tools: execute_cypher, execute_sql, get_schema
  • Agent can query any column, any relationship, any aggregation
  • Adding new data = no code changes
  • Fewer lines of code, more powerful system

Guardrail

The query guard (Phase 2) validates every generated query before execution. This is the safety mechanism.

3. Three-Layer Permission Guard (Not Tool-Based)

Decision

Permissions are enforced at three levels: agent awareness, query validation, and DB constraints. Agent never controls API keys or resource access.

Layer 1: Agent Awareness (Soft)

System prompt tells agent what it can access. Agent tries to comply voluntarily. Can fail—it's an LLM.

Layer 2: Query Guard (Medium)

Before execution: parse SQL/Cypher, validate columns/labels against permission set, reject violations. Uses sqlglot AST parsing for SQL.

Layer 3: DB-Level (Hard)

Read-only users, row policies, column grants. Physically impossible to bypass.

Why Not Tool-Based?

Tool-based permission checks scale poorly. A query guard that validates any query is simpler and more flexible. Permissions are modeled as data, not baked into tool definitions.

4. Single-Write Dual-Read Pattern

Decision

Neo4j is the source of truth (single write path). ClickHouse is read-only replicated view. Sync via direct inserts on write (Phase 1), Kafka later (Phase 5+).

Why

  • Simplifies data consistency (no merge conflicts)
  • Clear data lineage
  • Can always re-materialize ClickHouse from Neo4j if corrupted

5. Wide Event Table (Not Pre-Sharded Fact Tables)

Decision

Single ClickHouse table (~65 columns) for all events, not separate tables for campaigns, deals, leads, etc.

Why

  • Agent can query any dimension without knowing table structure
  • Columnar DB makes wide tables efficient
  • No schema migration when adding new dimension
  • Natural for denormalized fact tables

Tradeoff

Need careful denormalization and enrichment during sync to populate all columns.

Security & Permission Model

QueryPermissions Model

Every request is associated with a QueryPermissions object, resolved from the API key at request time:

@dataclass
class QueryPermissions:
    tenant_id: str                      # Which tenant owns this request
    allowed_labels: set[str] | None     # Graph node types (None = all)
    allowed_edge_types: set[str] | None # Relationships (None = all)
    allowed_event_categories: set[str]  # Event types (lifecycle, attribution, etc.)
    allowed_columns: set[str] | None    # ClickHouse columns (None = all)
    max_rows: int = 1000                # Query result cap
    read_only: bool = True              # Always true for agent

Key principle: The agent NEVER controls this. It's resolved by the API layer and passed immutably to the agent.

Three-Layer Permission Guard

Layer 1: Agent Awareness (Soft Guardrail)

System prompt tells the agent what it can access:

"You can access the following graph nodes: Campaign, Lead, Channel, Account.
You can query these ClickHouse event categories: attribution, lifecycle, subscription.
Available columns: campaign_id, campaign_name, lead_id, lead_status, deal_amount, etc.
..."

Why soft: It's an LLM; it might forget or misunderstand. We don't rely on this alone.

Layer 2: Query Guard (Medium Guardrail)

SQL Guard (ClickHouse)

Uses sqlglot to parse SQL into an AST and validate before execution:

  • Reject non-SELECT operations
  • Extract all referenced columns, validate against allowed_columns
  • Inject tenant_id filter (always, agent can't omit it)
  • Enforce event_category filter if restricted
  • Cap LIMIT to max_rows

Cypher Guard (Neo4j)

  • Reject all write keywords (CREATE, DELETE, DETACH, SET, REMOVE, MERGE, DROP)
  • Extract node labels from query, validate against allowed_labels
  • Extract relationship types, validate against allowed_edge_types
  • Inject tenant_id scoping if missing

Why medium: Catches 99% of violations, but sophisticated attacks might slip through (though db-level guard catches them).

Layer 3: Database-Level Enforcement (Hard Backstop)

ClickHouse

  • Agent connects via read-only user (separate from CRUD user)
  • Row policy enforced: CREATE ROW POLICY tenant_iso ON events USING tenant_id = currentUser()
  • Even if Layer 2 fails and agent queries without tenant_id, row policy blocks inter-tenant data leaks

Neo4j

  • Agent uses separate read-only connection (distinct from write connection)
  • DB driver-level enforcement prevents writes

Why hard: DB-enforced, can't be bypassed by application code.

Attack Scenarios

Attack Layer Caught Details
Agent generates non-SELECT query Layer 2 (Guard) Guard rejects DELETE, INSERT, DROP, etc.
Agent accesses forbidden column Layer 2 (Guard) Guard validates against allowed_columns
Agent queries without tenant_id filter Layer 3 (DB) Row policy filters rows by tenant
Agent accesses forbidden graph node Layer 2 (Guard) Guard validates labels in Cypher
Query result exceeds max_rows Layer 2 (Guard) Guard enforces LIMIT cap
Agent generates syntactically invalid SQL Layer 2 (Guard) Guard returns error, agent self-heals and retries

Self-Healing Error Flow

  1. Agent generates a query and calls a tool
  2. Guard validates it:
    • If permission error: "Column X not permitted" → Agent retries with different column
    • If syntax error: DB error message → Agent fixes syntax and retries
    • If valid: Query executes, results returned to agent
  3. Max 3 retries per query. If still failing, agent gives up and reports to user.
  4. User can refine their question and try again.

ClickHouse Events Table Schema

Single wide table, ~65 columns, MergeTree engine:

  • Core: event_id, tenant_id, event_type, event_category, occurred_at
  • Dimensions: campaign_*, channel_*, account_*, product_*, lead_*, deal_*, visitor_*, subscription_*, content_*, contact_*
  • Partitioned by month: PARTITION BY toYYYYMM(occurred_at)
  • Ordered by: (tenant_id, event_category, event_type, occurred_at)

Why wide? Agent can query any column independently without schema knowledge. Columnar DB makes it efficient.

Agent Tool Definitions

Tool 1: execute_cypher

@tool
async def execute_cypher(query: str, params: dict | None = None) -> list[dict]:
    """Run a read-only Cypher query against the Neo4j graph.
    Available node types: Lead, Contact, Deal, Campaign, Account, Visitor, 
                         Channel, Product, Content, EmailCampaign, Email, ProductAccount
    """
    safe_query = validate_and_scope_cypher(query, _current_permissions())
    return await graph.execute_read(safe_query, params or {})

Tool 2: execute_sql

@tool
async def execute_sql(query: str, params: dict | None = None) -> list[dict]:
    """Run a read-only SQL query against the ClickHouse events table.
    The events table has ~65 columns covering all GTM dimensions.
    """
    safe_query = validate_and_scope_sql(query, _current_permissions())
    return await ch.query(safe_query, params or {})

Tool 3: get_schema

@tool
async def get_schema() -> dict:
    """Returns the graph schema and ClickHouse columns.
    Use this to understand what data is available before writing queries.
    """
    return {
        "graph": await graph.get_schema(),
        "clickhouse_columns": EVENTS_TABLE_COLUMNS,
    }

System Prompt Structure

The agent's system prompt includes (generated dynamically):

  1. Role: "You are the gtmDB query planner..."
  2. Graph schema: Node types, edge types, properties
  3. ClickHouse schema: Event types, columns, data types
  4. Permission summary: "You may access: {allowed_labels}, {allowed_categories}..."
  5. Tool instructions: When to use Cypher (traversals), when to use SQL (aggregations)
  6. Examples: Sample questions and expected Cypher/SQL

Query Flow: End-to-End Example

User Question

"What are my best campaigns by LTV in the last 3 months, and show me the campaigns' content?"

Agent Steps

  1. Classify: "multi_engine" (needs SQL for LTV aggregation + Cypher for content graph traversal)
  2. Call execute_sql: Generate query to aggregate deal_amount by campaign from events table where occurred_at > 3 months ago
  3. Guard validates: Checks columns, injects tenant_id, enforces LIMIT
  4. Get results: Top 10 campaigns by LTV
  5. For each campaign: Call execute_cypher to traverse (Campaign)←[:CONTENT]-(Content) to fetch associated content
  6. Guard validates: Checks labels (Campaign, Content), injects tenant_id
  7. Combine results: Agent enriches campaign LTV data with content details in its reasoning
  8. Return: Structured answer with campaigns, LTV, and associated content

If Errors Occur

If Cypher syntax is invalid, guard returns error. Agent sees error, self-corrects, retries (max 3 times). If guard rejects for permission, agent acknowledges and tries a different approach.

Testing Strategy

  • Cypher guard: reject writes, validate labels
  • Permission model: resolve from API keys

Integration Tests

  • Write operations disguised in comments
